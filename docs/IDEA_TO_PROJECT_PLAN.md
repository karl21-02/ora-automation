# Idea-to-Project Pipeline (Phase G)

> 아이디어 검증을 위한 멀티 페르소나 실시간 토론 시스템

## 핵심 철학

### AI가 줄 수 없는 것 vs 줄 수 있는 것

| AI가 줄 수 없는 것 | AI가 줄 수 있는 것 |
|-------------------|-------------------|
| 아이디어 성공 예측 | **놓친 관점 제시** |
| 정확한 시장 규모 | **더 좋은 질문** |
| "이게 될 거야" 보장 | **검증해야 할 가설 명확화** |
| 점수/판정 | **빠른 반박 (Devil's Advocate)** |

**핵심 원칙**: AI는 답을 주지 않는다. 사용자가 스스로 깨닫게 돕는다.

---

## Ora다운 접근: 멀티 페르소나 토론

### 기존 설계 vs 새 접근

| 기존 (버림) | 새 접근 (채택) |
|-------------|---------------|
| AI가 점수를 줌 ("7.2점") | AI가 질문을 던짐 |
| 순차적 파이프라인 | 실시간 토론 관찰 |
| 30분 후 리포트 | 즉시 다각도 피드백 |
| MVP "정의" | MVP "가설" |
| Go/No-Go 판정 | 검증해야 할 질문 목록 |

### 핵심 컨셉

```
┌─────────────────────────────────────────────────────────────────┐
│                    아이디어 토론장 (Arena)                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  사용자가 아이디어를 던지면                                      │
│  → 여러 페르소나가 각자의 관점에서 즉시 반응                     │
│  → 사용자는 토론을 "관찰"하며 인사이트 획득                      │
│  → 최종 output은 "점수"가 아니라 "검증 질문"                    │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 페르소나 구성

### 5개 핵심 페르소나

| 페르소나 | 관점 | 핵심 질문 |
|----------|------|-----------|
| 🎯 **PotentialCustomer** | 실제 사용자 | "이거 왜 써야 해? 지금 뭘로 해결하고 있는데?" |
| 💰 **Investor** | 투자/수익 | "시장 규모는? 경쟁자 대비 moat는?" |
| 🔧 **TechRealist** | 기술 현실 | "진짜 만들 수 있어? 숨은 복잡도는?" |
| 🏃 **Competitor** | 경쟁자 시점 | "우리가 따라하면? 무료로 풀면?" |
| 😈 **Skeptic** | 근본적 의심 | "이게 진짜 문제야? 더 큰 문제는 없어?" |

### 페르소나 특성

```yaml
PotentialCustomer:
  perspective: "실제로 돈 내고 쓸 사람"
  questions:
    - "지금 이 문제를 어떻게 해결하고 있나요?"
    - "기존 솔루션의 불만은 뭔가요?"
    - "이 제품에 월 얼마까지 낼 의향이 있나요?"
  challenge_style: "현실적 사용 시나리오 기반"

Investor:
  perspective: "돈을 투자할 사람"
  questions:
    - "TAM/SAM/SOM은?"
    - "왜 지금이 적기인가?"
    - "팀이 이걸 할 수 있는 이유는?"
  challenge_style: "숫자와 논리로 압박"

TechRealist:
  perspective: "실제로 만들 개발자"
  questions:
    - "핵심 기술 리스크는?"
    - "MVP 실제 개발 기간은?"
    - "스케일링 병목은 어디?"
  challenge_style: "구현 현실성 검증"

Competitor:
  perspective: "시장의 기존 플레이어"
  questions:
    - "우리가 3개월 안에 똑같이 만들면?"
    - "우리가 가격을 절반으로 내리면?"
    - "우리 기존 고객이 왜 이탈해야 해?"
  challenge_style: "경쟁 시뮬레이션"

Skeptic:
  perspective: "근본적 의심가"
  questions:
    - "이게 진짜 해결할 가치가 있는 문제야?"
    - "10년 후에도 이 문제가 존재할까?"
    - "더 근본적인 해결책은 없어?"
  challenge_style: "전제 자체를 흔듦"
```

## 사용자 경험 플로우

### Step 1: 아이디어 입력

```

├─────────────────────────────────────────────────────────────────┤
│  💰 Investor:                                                    │
│  "Otter.ai가 $1B 밸류에이션인데, 영어 시장 중심이죠.             │
│   한국어만으로 어떤 규모의 비즈니스가 가능하다고 보세요?         │
│   글로벌 확장 계획은요?"                                        │
├─────────────────────────────────────────────────────────────────┤
│  🔧 TechRealist:                                                 │
│  "Whisper 한국어 정확도가 85% 정도인데,                          │
│   여기서 액션아이템을 추출하면 정확도가 더 떨어질 거예요.        │
│   '액션아이템 추출 정확도 80%' 어떻게 달성할 건가요?"           │
├─────────────────────────────────────────────────────────────────┤
│  🏃 Competitor (as Clova):                                       │

│   저희가 같은 기능 추가하면 어떻게 대응하실 건가요?"            │

│  "잠깐, 회의록이 진짜 문제인가요?                                │

└─────────────────────────────────────────────────────────────────┘
```

```
├─────────────────────────────────────────────────────────────────┤
```

### 데이터 모델

```python
@dataclass
class IdeaInput:
    """사용자 아이디어 입력."""
    raw_text: str
    context: dict = field(default_factory=dict)


@dataclass
class PersonaResponse:
    """단일 페르소나의 응답."""
    persona_id: str           # "customer", "investor", etc.
    persona_name: str         # "PotentialCustomer"
    perspective: str          # 관점 설명
    challenge: str            # 던지는 질문/도전
    underlying_concern: str   # 근본적 우려
    follow_up_if: str         # "만약 ~라면 다음 질문은..."


@dataclass
class ArenaRound:
    """토론 한 라운드."""
    round_num: int
    user_input: str
    persona_responses: list[PersonaResponse]
    key_tensions: list[str]   # 드러난 핵심 긴장점


@dataclass
class ValidationQuestion:
    """검증해야 할 질문."""
    question: str
    category: str             # "customer", "tech", "market", "competition"
    source_persona: str       # 어느 페르소나에서 도출됐는지
    validation_method: str    # 검증 방법
    effort_level: str         # "1_hour", "1_day", "1_week"


@dataclass
class ImmediateAction:
    """지금 바로 할 수 있는 행동."""
    action: str
    expected_outcome: str
    time_required: str


@dataclass
class ArenaResult:
    """전체 토론 결과."""
    idea_input: IdeaInput
    rounds: list[ArenaRound]
    validation_questions: list[ValidationQuestion]
    immediate_actions: list[ImmediateAction]
    key_risks: list[str]
    key_opportunities: list[str]
```

### API 설계

```python
# idea_arena_router.py

# 1. 아이디어 토론 시작 (SSE 스트리밍)
POST /api/v1/arena/start
Body: { "idea": "음성 회의록 서비스" }
Response: SSE stream of PersonaResponse

# 2. 사용자 응답 추가 (토론 계속)
POST /api/v1/arena/{session_id}/respond
Body: { "response": "B2B 기업용으로 갈 거예요" }
Response: SSE stream of PersonaResponse

# 3. 검증 질문 도출
POST /api/v1/arena/{session_id}/extract-questions
Response: { validation_questions, immediate_actions }

# 4. 세션 조회
GET /api/v1/arena/{session_id}
Response: ArenaResult
```

## 성공 지표

| 지표 | 목표 | 측정 방법 |
|------|------|-----------|
| 첫 응답 시간 | < 3초 | 아이디어 입력 → 첫 페르소나 응답 |
| 인사이트 발견율 | > 80% | "몰랐던 관점을 알게 됐다" 응답 비율 |
| 검증 행동 전환율 | > 50% | 즉시 행동 중 1개 이상 실행한 비율 |
| 재사용율 | > 60% | 다른 아이디어로 다시 사용한 비율 |

## 핵심 차별점

```
❌ 기존 아이디어 검증 도구들:
   - "점수"를 줌 (의미 없음)
   - 체크리스트 기반 (하드코딩)
   - 결과 리포트 (읽고 끝)

✅ Ora Idea Arena:
   - 질문을 던짐 (Socratic Method)
   - LLM 기반 동적 반응
   - 실시간 토론 (인터랙티브)
   - 즉시 행동으로 연결
```
